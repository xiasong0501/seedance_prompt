# `series_pipeline.local.json` 预处理参数理解

这组参数位于：

- `config/series_pipeline.local.json`
- `preprocess`

它们都只作用在“本地预处理”阶段，也就是视频送进大模型之前的那一层。

完整链路大致是：

1. `ffmpeg` 抽音频
2. `faster-whisper` 做 ASR 语音转写
3. `PySceneDetect` 做切镜
4. 根据 `max_keyframes` 选出要保留的关键场景
5. 只在这些已选关键场景内做 OCR 取样
6. `RapidOCR` 做字幕 / 画面文字识别
7. 抽取关键帧
8. 额外生成给模型使用的缩放版关键帧
9. 把 OCR 结果再关联回关键帧
10. 再把 transcript / OCR / keyframes 交给大模型做视频理解和剧本重建

## 当前配置

```json
"preprocess": {
  "asr_model_size": "small",
  "asr_language": "zh",
  "asr_device": "auto",
  "asr_compute_type": "auto",
  "asr_beam_size": 5,
  "shot_threshold": 27.0,
  "shot_min_scene_len": 15,
  "max_keyframes": null,
  "ocr_crop_bottom_ratio": 0.35,
  "ocr_sample_interval_seconds": 0.5,
  "keyframe_scale_ratio": 0.5
}
```

## 参数说明

### `asr_model_size`

作用：

- 控制 `faster-whisper` 用哪个 ASR 模型大小。

理解：

- 模型越大，通常识别越准，但越慢，也更吃显存或内存。

当前值：

- `small`
- 属于速度和质量比较平衡的档位。

怎么调：

- 想更快：可以尝试 `tiny` 或 `base`
- 想更准：可以尝试 `medium`

---

### `asr_language`

作用：

- 指定 ASR 识别语言。

当前值：

- `zh`
- 表示按中文识别。

怎么调：

- 主要是中文视频就保持 `zh`
- 如果后面遇到大量英文或多语混杂视频，再考虑改策略

---

### `asr_device`

作用：

- 控制 ASR 跑在 GPU 还是 CPU 上。

常见取值：

- `auto`
- `cuda`
- `cpu`

当前值：

- `auto`

当前行为：

- 系统会优先尝试 GPU
- 如果 GPU 不可用，再自动回退到 CPU

适合场景：

- 日常推荐保持 `auto`
- 如果你明确知道机器 GPU 可用，也可以强制写成 `cuda`

---

### `asr_compute_type`

作用：

- 控制 ASR 推理精度和量化方式。

常见理解：

- `float16`
  适合 GPU，速度快，质量稳
- `int8_float16`
  GPU 上更省资源
- `int8`
  更省资源，适合 CPU
- `auto`
  让系统自己选

当前值：

- `auto`

当前行为：

- 会和 `asr_device` 一起决定最终使用的推理方式

建议：

- 不熟悉时保持 `auto`

---

### `asr_beam_size`

作用：

- 控制 ASR 解码时 beam search 的宽度。

理解：

- 值越大，通常更稳更准，但也更慢。

当前值：

- `5`

建议：

- `5` 是比较常用的平衡值
- 如果只想提速，可以降到 `3`
- 如果极致追求准确率，可以试着升一点，但耗时会增加

---

### `shot_threshold`

作用：

- 控制镜头切分灵敏度。

理解：

- 值越低，越容易判定为“切镜”，scene 数会更多
- 值越高，越保守，scene 数会更少

当前值：

- `27.0`

建议：

- 想抓更多快速切换镜头：适当降低
- 想减少切得太碎：适当提高

---

### `shot_min_scene_len`

作用：

- 控制两次切镜之间允许的最小长度。

注意：

- 这里更接近“最少帧数约束”，不是秒数。

理解：

- 值越大，越不容易把很短的画面变化也切成独立 scene
- 值越小，越容易把快速切换拆出来

当前值：

- `15`

建议：

- 觉得 scene 太碎，可以加大
- 觉得很多快节奏切换被漏掉，可以减小

---

### `max_keyframes`

作用：

- 控制最多保留多少个 scene 来抽关键帧。

当前值：

- `null`

当前行为：

- 不限制数量
- 检测到多少个 scene，就抽多少张关键帧

影响：

- 关键帧越多，后面大模型看到的画面证据越多
- 但也会明显增加处理时间和 token 消耗

建议：

- 追求理解质量时，`null` 是合理的
- 如果后面想省时间、省 token，可以改成一个固定值，例如 `40`、`60`

---

### `ocr_crop_bottom_ratio`

作用：

- 控制 OCR 只看画面底部多少比例。

当前值：

- `0.35`

当前行为：

- 只截取底部 35% 区域做 OCR
- 主要是为了抓字幕，而不是整个画面所有文字

理解：

- 值越大，OCR 看到的画面区域越高
- 值越小，越聚焦字幕底栏

建议：

- 如果字幕基本都在底部，`0.35` 很合适
- 如果很多关键信息出现在画面中下部，可以考虑调到 `0.45` 或 `0.5`

---

### `ocr_sample_interval_seconds`

作用：

- 控制 OCR 在“已选关键场景”里的取样频率。

当前值：

- `0.5`

当前行为：

- 不再对整部视频全量密采样
- 只会在已经选中的关键场景里抽 OCR 取样帧
- 后面再把 OCR 结果关联回对应关键帧

理解：

- 值越小，OCR 覆盖越密，更不容易漏字幕
- 但 OCR 的时间开销也会增加

建议：

- 如果你的视频字幕切换很快，`0.5` 很稳
- 如果你更在意速度，可以试 `0.75` 或 `1.0`

---

### `keyframe_scale_ratio`

作用：

- 控制关键帧输出时的缩放比例。

当前值：

- `0.5`

当前行为：

- 原始关键帧会先写到 `preprocess/frames/`
- 模型实际使用的缩放图会写到 `preprocess/frames_for_model/`
- 缩放完成后，原始关键帧目录会被立即清理，避免后续误用高分辨率图
- 关键帧宽高都会乘以 `0.5`
- 这会明显减小图片文件体积
- 对 OpenAI 路线，配合 `image_detail=low` 时主要是减少上传体积和部分延迟

理解：

- 值越小，关键帧越省空间、越轻
- 但过小可能损失细节

建议：

- `0.5` 是比较稳的压缩比例
- 如果你担心细节丢失，可以改成 `0.75`
- 如果后面切到高 detail 模式，更高分辨率会更有意义

运行反馈：

- 预处理完成后，终端会打印：
  - `scale_ratio`
  - 一个示例分辨率，例如 `720x1280 -> 360x640`
  - 原始关键帧目录
  - 模型输入图目录
- 这些信息也会写进：
  - `analysis/<剧名>/<epXX>/preprocess/preprocess_manifest.json`
  - 对应字段是 `keyframe_resize_summary`
- 另外：
  - `preprocess/ocr_samples/` 会在 OCR 结束后立即清理
  - `preprocess/frames/` 会在生成完 `frames_for_model/` 后立即清理
  - 因此后续真正留给模型和流程使用的只有 `frames_for_model/`

## 这套当前配置的整体含义

这组配置现在的整体风格是：

- ASR：中文、小模型、优先 GPU、beam=5
- 切镜：中等灵敏度
- 关键帧：可缩放，兼顾画面证据和输入体积
- OCR：只在已选关键场景内取样，再重点抓底部字幕

也就是说，它更偏“质量优先、信息尽量保留”，而不是极致省时间和省 token。

## 实用调参建议

### 1. 想更快

- 优先减少 `max_keyframes`
- 适当提高 `shot_threshold`
- 必要时把 `asr_model_size` 降低

### 2. 想更准

- 优先把 `asr_model_size` 提高到 `medium`
- 保持 `max_keyframes = null`
- 必要时把 `ocr_crop_bottom_ratio` 调大一点

### 3. 想减少 token 消耗

- 优先控制关键帧数量
- 因为后面大模型最吃 token 的通常不是 ASR，而是“喂进去的 keyframes 太多”

## 一句话总结

这组参数里：

- `asr_*` 决定语音转写质量和速度
- `shot_*` 决定切镜多少
- `max_keyframes` 决定后面给大模型多少画面证据
- `ocr_crop_bottom_ratio` 决定 OCR 更偏字幕还是更偏整画面文字
- `ocr_sample_interval_seconds` 决定 OCR 是否会漏掉快速切换字幕
- `keyframe_scale_ratio` 决定模型输入关键帧的缩放程度，同时保留原始关键帧供人工检查

## 额外说明：OpenAI 图片 token 优化

除了 `preprocess` 里的参数，OpenAI 路线还有一个很重要的 provider 参数：

- `providers.openai.image_detail`

常见取值：

- `low`
- `auto`
- `high`

当前推荐：

- 如果 OCR 已经把字幕补得比较完整，OpenAI 路线建议优先用 `low`

理解：

- `low` 模式下，OpenAI 图片输入会按固定基础 token 成本计费
- 这时缩小关键帧分辨率，主要是减少图片文件体积和上传延迟
- 如果你改成 `auto` 或 `high`，图片分辨率就会更明显影响图片 token 成本
